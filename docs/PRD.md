## 1. 핵심 정보

### 서비스 목적

사용자들이 7일·14일·30일 단기 챌린지에 참여하고 매일 이미지 기반 인증을 통해 목표를 달성하도록 돕는 모바일 퍼스트 챌린지 플랫폼이다.

### 주요 사용자 유형

| 유형              | 설명                                             | 권한                     |
| ----------------- | ------------------------------------------------ | ------------------------ |
| **챌린지 운영자** | 챌린지를 직접 생성·관리하는 일반 사용자          | 챌린지 CRUD, 참여자 관리 |
| **챌린지 참여자** | 다른 운영자가 만든 챌린지에 참여하는 일반 사용자 | 챌린지 참여, 인증 업로드 |
| **플랫폼 관리자** | 전체 서비스를 관리하는 내부 운영자               | 전체 데이터 조회·관리    |

> [가정] 일반 사용자는 운영자와 참여자 역할을 동시에 수행할 수 있다. 예: A 챌린지의 운영자이면서 B 챌린지의 참여자.

### MVP 범위

**MVP에 포함되는 것**

- Google OAuth 로그인/로그아웃
- 챌린지 생성, 수정, 삭제 (운영자)
- 챌린지 목록 조회, 참여 신청
- 매일 이미지 기반 인증(check-in) 업로드
- 내 챌린지 및 인증 기록 조회
- 실시간 참여자 현황 업데이트 (Supabase Realtime)
- 관리자 대시보드 (챌린지·사용자·통계 관리)

**MVP에 포함되지 않는 것**

- 소셜 기능 (댓글, 좋아요, 팔로우)
- 알림 시스템 (이메일, 푸시)
- Streak 시스템 및 배지
- 랭킹 및 리더보드
- 결제 및 유료 챌린지
- 모바일 앱 (iOS/Android)

## 2. 사용자 여정

### 2-1. 챌린지 운영자 여정

```
1. 랜딩 페이지 진입
   ↓ (구글 로그인 버튼 클릭)
2. Google OAuth 인증
   ↓ (인증 성공 → 자동 리디렉션)
3. 홈 페이지 (로그인 완료 상태)
   ↓ (새 챌린지 만들기 클릭)
4. 챌린지 생성 페이지
   → 제목, 설명, 기간(7/14/21/30일), 카테고리, 인증 방법, 최대 참여자 수, 썸네일 입력
   ↓ (생성 완료)
5. 챌린지 상세 페이지 (내 챌린지)
   → 참여자 목록 실시간 확인
   → 챌린지 정보 수정 버튼
   → 챌린지 종료/삭제 버튼
   ↓ (챌린지 기간 동안)
6. 매일 인증 업로드 (운영자도 참여자로서 인증 가능)
   ↓ (챌린지 종료 후)
7. 내 챌린지 목록에서 종료된 챌린지 확인
```

**주요 예외 케이스**

- 챌린지 생성 후 참여자가 0명인 경우: 내 챌린지 목록에 "참여자 없음" 표시
- 챌린지 기간 중 수정 시: 제목·설명만 수정 가능, 기간·최대 참여자 수는 변경 불가

### 2-2. 챌린지 참여자 여정

```
1. 랜딩 페이지 진입
   ↓ (구글 로그인)
2. 홈 페이지
   ↓ (챌린지 목록 탐색)
3. 챌린지 목록 페이지
   → 카테고리 필터, 기간 필터로 원하는 챌린지 탐색
   ↓ (챌린지 카드 클릭)
4. 챌린지 상세 페이지
   → 챌린지 정보 확인 (기간, 설명, 참여자 수, 인증 방법)
   → 참여하기 버튼 클릭
   ↓ (참여 신청 완료)
5. 내 챌린지 목록에 추가됨
   ↓ (당일 인증 버튼 클릭)
6. 인증 업로드 페이지
   → 이미지 업로드 (카메라/갤러리 선택)
   → 인증 메모 작성 (선택)
   → 제출
   ↓ (인증 완료)
7. 챌린지 상세 페이지로 복귀
   → 오늘 인증 완료 표시
   → 전체 인증 현황 확인
   ↓ (다음날)
8. 반복: 인증 업로드 (챌린지 기간 종료까지)
   ↓ (챌린지 종료)
9. 내 인증 기록 페이지에서 전체 인증 이력 조회
```

**주요 예외 케이스**

- 최대 참여자 수 초과 시: "참여 마감" 표시, 참여 버튼 비활성화
- 당일 인증 완료 후 재시도: "오늘은 이미 인증하셨습니다" 메시지, 재업로드 불가 (MVP)
- 챌린지 탈퇴: 탈퇴 시 기존 인증 기록은 유지

### 2-3. 플랫폼 관리자 여정

```
1. 관리자 로그인 페이지 (/admin/login) 진입
   ↓ (Google OAuth 로그인 — 관리자 전용 콜백)
2. 관리자 전용 OAuth 콜백 (/admin/auth/callback)
   → role='admin' 확인 후 대시보드로 이동
   → role!='admin'이면 /admin/login으로 리디렉션 (에러 메시지)
   ↓ (관리자 계정 확인 완료)
3. 관리자 대시보드
   → 전체 서비스 현황 (MAU, 챌린지 수, 오늘 인증 수) 확인
   ↓ (메뉴 선택)
3a. 챌린지 관리 페이지
    → 전체 챌린지 목록 조회
    → 부적절한 챌린지 강제 종료/삭제
    → 챌린지 상태 변경 (진행중/종료/숨김)
3b. 사용자 관리 페이지
    → 전체 사용자 목록 조회
    → 사용자 활동 내역 확인
    → 부적절한 사용자 비활성화
3c. 통계 페이지
    → 기간별 신규 가입자 추이
    → 챌린지 생성 및 참여 현황
    → 인증 업로드 현황
    → Recharts 기반 차트로 시각화
```

## 3. 기능 명세

### 3-1. 인증 기능

| ID    | 기능명                     | 설명                                                                                            | MVP 필수 | 관련 페이지                       |
| ----- | -------------------------- | ----------------------------------------------------------------------------------------------- | -------- | --------------------------------- |
| F001  | 서비스 Google OAuth 로그인 | Supabase Auth를 통한 Google 소셜 로그인. role='user'인 일반 사용자만 허용                       | 필수     | 랜딩 페이지, 서비스 로그인 페이지 |
| F001A | 관리자 Google OAuth 로그인 | 관리자 전용 Google 로그인. role='admin'인 사용자만 허용. 독립된 로그인 페이지와 OAuth 콜백 사용 | 필수     | 관리자 로그인 페이지              |
| F002  | 로그아웃                   | 현재 세션 종료. 서비스는 랜딩 페이지, 관리자는 관리자 로그인 페이지로 리디렉션                  | 필수     | 서비스 헤더, 관리자 사이드바      |

**F001 상세 (서비스 로그인)**

- 동작 설명: 일반 사용자가 "Google로 로그인" 버튼 클릭 시 Google OAuth 실행, `/auth/callback`으로 콜백 처리 후 홈 페이지로 이동
- 입력: Google 계정 자격증명
- 출력: Supabase 세션(JWT), users 테이블 레코드 생성 또는 업데이트
- 비즈니스 규칙:
  - 동일 Google 계정으로 재로그인 시 Supabase Auth의 `auth.users.last_sign_in_at`이 자동 업데이트됨
  - role='admin'인 계정으로 서비스 로그인 시도 시: 세션을 즉시 종료하고 로그인 페이지에 "관리자 계정은 관리자 페이지에서 로그인해주세요" 에러 메시지 표시
- 엣지 케이스:
  - Google 인증 실패 시: 로그인 페이지에 에러 메시지 표시
  - 팝업 차단 시: 동일 탭 내 리디렉션 방식으로 폴백

**F001A 상세 (관리자 로그인)**

- 동작 설명: 관리자가 `/admin/login` 페이지에서 "Google 계정으로 로그인" 버튼 클릭 시 Google OAuth 실행, `/admin/auth/callback`으로 콜백 처리 후 관리자 대시보드로 이동
- 입력: Google 계정 자격증명 (role='admin'이 부여된 계정)
- 출력: Supabase 세션(JWT)
- 비즈니스 규칙:
  - role='admin'이 아닌 계정으로 관리자 로그인 시도 시: 세션을 즉시 종료하고 관리자 로그인 페이지에 "관리자 계정이 아닙니다" 에러 메시지 표시
  - 서비스 로그인과 완전히 독립된 OAuth 콜백 경로 사용 (`/admin/auth/callback`)
- 엣지 케이스:
  - Google 인증 실패 시: 관리자 로그인 페이지에 에러 메시지 표시
- UI 특성:
  - 데스크톱 퍼스트 디자인 (서비스의 모바일 430px 레이아웃과 독립)
  - 사이드바 없는 단독 페이지 (로그인 후 사이드바 포함 대시보드로 이동)

### 3-2. 챌린지 관리 기능 (운영자)

| ID   | 기능명           | 설명                                                                                          | MVP 필수 | 관련 페이지                    |
| ---- | ---------------- | --------------------------------------------------------------------------------------------- | -------- | ------------------------------ |
| F003 | 챌린지 생성      | 운영자가 새 챌린지를 생성. 제목, 설명, 기간, 카테고리, 인증 방법, 최대 참여자 수, 썸네일 설정 | 필수     | 챌린지 생성 페이지             |
| F004 | 챌린지 수정      | 챌린지 기본 정보(제목, 설명) 수정. 진행 중에는 기간/참여자 수 변경 불가                       | 필수     | 챌린지 생성 페이지 (수정 모드) |
| F005 | 챌린지 삭제      | 운영자가 챌린지 삭제. 참여자가 1명 이상이면 삭제 불가, 종료 처리만 가능                       | 필수     | 챌린지 상세 페이지             |
| F006 | 챌린지 상태 관리 | 챌린지 상태를 모집중/진행중/종료로 전환. start_date/end_date 도래 시 자동 전환                | 필수     | 챌린지 상세 페이지             |

**F006 상세**

- 동작 설명: 챌린지 상태를 수동 또는 자동으로 전환
- 수동 전환 (운영자): 챌린지 상세 페이지에서 운영자가 직접 상태 변경 (종료/숨김 처리)
- 자동 전환 (스케줄러):
  - `recruiting` → `in_progress`: start_date 당일 KST 00:00 도래 시 자동 전환
  - `in_progress` → `finished`: end_date 익일 KST 00:00 도래 시 자동 전환
  - 구현 방식: Supabase pg_cron 확장을 활용하여 매일 KST 00:00(UTC 15:00)에 상태 전환 쿼리 실행
- 비즈니스 규칙:
  - 운영자는 `recruiting` → `finished`(강제 종료) 또는 `hidden`으로만 수동 전환 가능
  - 상태가 `finished`인 챌린지는 다시 `recruiting`/`in_progress`로 되돌릴 수 없음
  - 관리자는 모든 상태로 변경 가능

**F003 상세**

- 동작 설명: 운영자가 폼을 작성하여 챌린지 생성. 썸네일 이미지는 Supabase Storage에 업로드
- 입력:
  - `title`: 챌린지 제목 (2~50자)
  - `description`: 챌린지 설명 (10~500자)
  - `duration_days`: 기간 선택 (7 | 14 | 21 | 30)
  - `category`: 카테고리 (건강/운동 | 학습 | 생활습관 | 기타)
  - `check_in_description`: 인증 방법 설명 (10~200자)
  - `max_participants`: 최대 참여자 수 (1~100, 기본값 30)
  - `thumbnail_url`: 썸네일 이미지 (선택, JPG/PNG/WebP, 최대 5MB)
- 출력: challenges 테이블 레코드 생성, 생성된 챌린지 상세 페이지로 이동
- 비즈니스 규칙:
  - 챌린지 생성 시 status는 'recruiting'(모집중)으로 시작
  - start_date는 생성일 기준 익일부터 선택 가능
  - 운영자 본인은 자동으로 참여자로 추가되지 않음 (별도 참여 신청 필요)
  - 썸네일 이미지는 업로드 전 클라이언트에서 리사이징 처리: 최대 1920px, WebP 변환, 품질 80% 압축. Supabase Storage Image Transformation을 통해 조회 시 자동 리사이징 가능 (`width`, `height` 파라미터 활용)
- 엣지 케이스:
  - 이미지 업로드 실패 시: 썸네일 없이 챌린지 생성 진행 (기본 이미지 사용)
  - 필수 필드 누락 시: 해당 필드에 에러 메시지 표시, 폼 제출 차단

### 3-3. 챌린지 탐색 및 참여 기능 (참여자)

| ID   | 기능명           | 설명                                                                   | MVP 필수 | 관련 페이지        |
| ---- | ---------------- | ---------------------------------------------------------------------- | -------- | ------------------ |
| F007 | 챌린지 목록 조회 | 공개 챌린지 목록. 카테고리·기간·상태 필터, 페이지네이션(무한 스크롤)   | 필수     | 챌린지 목록 페이지 |
| F008 | 챌린지 상세 조회 | 챌린지 상세 정보, 참여자 수, 인증 현황 조회. 실시간 참여자 수 업데이트 | 필수     | 챌린지 상세 페이지 |
| F009 | 챌린지 참여 신청 | 원하는 챌린지에 참여 신청. challenge_participants 테이블에 레코드 추가 | 필수     | 챌린지 상세 페이지 |
| F010 | 챌린지 탈퇴      | 참여 중인 챌린지에서 탈퇴. 기존 인증 기록 유지                         | 필수     | 챌린지 상세 페이지 |

**F007 상세**

- 동작 설명: 로그인 여부와 무관하게 챌린지 목록 조회 가능(비로그인 시 참여 불가)
- 입력: 필터(카테고리, 기간, 상태), 정렬(최신순 | 참여자 많은순 | 마감 임박순)
- 출력: 페이지당 12개 챌린지 카드 목록 (무한 스크롤)
- 비즈니스 규칙:
  - 상태가 'hidden'인 챌린지는 목록에서 제외
  - 종료된 챌린지는 기본 필터에서 제외 (필터로 포함 가능)
- 엣지 케이스:
  - 검색 결과 없음: "조건에 맞는 챌린지가 없습니다" 빈 상태 화면 표시

**F009 상세**

- 동작 설명: 로그인한 사용자가 참여하기 버튼 클릭 시 즉시 참여 완료
- 입력: challenge_id, user_id
- 출력: challenge_participants 테이블 레코드 생성, 챌린지 상세 페이지 참여자 수 실시간 반영
- 비즈니스 규칙:
  - 이미 참여 중인 챌린지는 재참여 불가
  - 최대 참여자 수 도달 시 참여 불가
  - 종료된 챌린지는 참여 불가
  - 비로그인 사용자는 클릭 시 로그인 페이지로 리디렉션
- 엣지 케이스:
  - 동시 참여 신청으로 최대 인원 초과 시: DB 트랜잭션으로 정합성 보장, 초과 요청자에게 "마감되었습니다" 메시지

**F010 상세**

- 동작 설명: 참여 중인 챌린지에서 탈퇴. `left_at` 필드를 현재 시각으로 업데이트하여 soft delete 처리
- 입력: challenge_id, user_id
- 출력: challenge_participants 레코드의 left_at 업데이트, 챌린지 참여자 수 감소
- 비즈니스 규칙:
  - 탈퇴 후 기존 인증 기록(challenge_checkins)은 그대로 유지됨
  - 탈퇴 후 재참여는 허용됨. 재참여 시 새 레코드를 생성 (기존 레코드의 left_at이 채워져 있으므로 Partial UNIQUE 인덱스 조건을 충족)
  - 재참여 시 이전 인증 기록도 그대로 조회됨 (challenge_id, user_id 기준)
- 엣지 케이스:
  - 종료된 챌린지에서 탈퇴 시도: 정상 처리 (이미 종료된 챌린지이므로 참여자 수 변동 없음)

### 3-4. 인증(Check-in) 기능

| ID   | 기능명              | 설명                                                       | MVP 필수 | 관련 페이지         |
| ---- | ------------------- | ---------------------------------------------------------- | -------- | ------------------- |
| F011 | 인증 업로드         | 오늘의 인증 이미지와 메모를 업로드. 하루 1회 제한          | 필수     | 인증 업로드 페이지  |
| F012 | 내 인증 기록 조회   | 챌린지별 나의 인증 이력 조회. 달력 뷰 형태                 | 필수     | 내 인증 기록 페이지 |
| F013 | 전체 인증 현황 조회 | 챌린지 상세 페이지에서 모든 참여자의 인증 현황 실시간 조회 | 필수     | 챌린지 상세 페이지  |

**F011 상세**

- 동작 설명: 참여자가 오늘의 인증 이미지를 촬영하거나 갤러리에서 선택하여 업로드
- 입력:
  - `image`: 인증 이미지 (JPG/PNG/WebP, 최대 10MB)
  - `memo`: 인증 메모 (선택, 최대 200자)
  - `challenge_id`: 해당 챌린지 ID
  - `check_in_date`: 인증 날짜 (기본값: 오늘 날짜, 한국 시간 기준)
- 출력: Supabase Storage에 이미지 저장, challenge_checkins 테이블 레코드 생성
- 비즈니스 규칙:
  - 당일(한국 시간 KST 00:00~23:59) 1회만 업로드 가능
  - check_in_date는 서버 UTC 기준이 아닌 KST(UTC+9) 기준 날짜로 저장. 클라이언트에서 `Intl.DateTimeFormat`을 사용하여 KST 기준 날짜(YYYY-MM-DD)를 계산 후 전송
  - 챌린지 기간 이외의 날짜에는 업로드 불가
  - 챌린지에 참여 중인 사용자만 업로드 가능
  - 이미지 파일만 허용 (MIME type 검증)
  - 인증 이미지는 업로드 전 클라이언트에서 리사이징 처리: 최대 1920px, WebP 변환, 품질 80% 압축
- 엣지 케이스:
  - 이미지 업로드 실패 시: 업로드 재시도 버튼 표시, DB 레코드 미생성
  - 파일 용량 초과 시: "10MB 이하의 이미지만 업로드 가능합니다" 에러 메시지

### 6-1. 테이블 설계

#### users 테이블

Supabase Auth의 `auth.users`를 참조하는 공개 프로필 테이블이다.

```typescript
type User = {
	id: string; // UUID, auth.users.id 참조, PK
	email: string; // 이메일 (Google 계정)
	nickname: string; // 표시 이름 (Google 계정 displayName 기본값)
	avatar_url: string | null; // 프로필 이미지 URL (Google 프로필 이미지)
	role: "user" | "admin"; // 사용자 역할, 기본값 'user'
	is_active: boolean; // 계정 활성화 여부, 기본값 true
	created_at: string; // ISO 8601 타임스탬프
	updated_at: string; // ISO 8601 타임스탬프
};
```

| 필드       | 타입        | 제약조건                                                    | 설명                    |
| ---------- | ----------- | ----------------------------------------------------------- | ----------------------- |
| id         | UUID        | PK, FK → auth.users(id) ON DELETE CASCADE                   | Supabase Auth 사용자 ID |
| email      | TEXT        | NOT NULL, UNIQUE                                            | 구글 계정 이메일        |
| nickname   | TEXT        | NOT NULL                                                    | 표시 이름               |
| avatar_url | TEXT        | NULLABLE                                                    | 프로필 이미지 URL       |
| role       | TEXT        | NOT NULL, DEFAULT 'user', CHECK (role IN ('user', 'admin')) | 사용자 역할             |
| is_active  | BOOLEAN     | NOT NULL, DEFAULT TRUE                                      | 계정 활성화 여부        |

#### challenge_participants 테이블

```typescript
type ChallengeParticipant = {
	id: string; // UUID, PK
	challenge_id: string; // FK → challenges.id
	user_id: string; // FK → users.id
	joined_at: string; // 참여 신청 시각
	left_at: string | null; // 탈퇴 시각 (null이면 현재 참여 중)
};
```

| 필드         | 타입        | 제약조건                                        | 설명           |
| ------------ | ----------- | ----------------------------------------------- | -------------- |
| id           | UUID        | PK, DEFAULT gen_random_uuid()                   | 참여 레코드 ID |
| challenge_id | UUID        | NOT NULL, FK → challenges(id) ON DELETE CASCADE | 챌린지 ID      |
| user_id      | UUID        | NOT NULL, FK → users(id) ON DELETE CASCADE      | 참여자 ID      |
| joined_at    | TIMESTAMPTZ | NOT NULL, DEFAULT NOW()                         | 참여 시각      |
| left_at      | TIMESTAMPTZ | NULLABLE, DEFAULT NULL                          | 탈퇴 시각      |

- Partial UNIQUE 인덱스: `UNIQUE(challenge_id, user_id) WHERE left_at IS NULL` — 현재 참여 중인 레코드에 한하여 중복 참여 방지. left_at이 채워진(탈퇴) 레코드는 유니크 조건 대상에서 제외되므로 탈퇴 후 재참여가 가능 (새 레코드 생성)
- 현재 참여 중 조건: `left_at IS NULL`
- 재참여 시나리오: 탈퇴(left_at 갱신) 후 재참여 신청 시 새 레코드가 INSERT됨. 이전 인증 기록은 (challenge_id, user_id) 기준으로 계속 조회 가능

### 6-2. 관계 정의

```
users (1) ──── (N) challenges                [운영자 1명이 여러 챌린지 생성]
users (1) ──── (N) challenge_participants    [사용자 1명이 여러 챌린지 참여]
challenges (1) ─── (N) challenge_participants [챌린지 1개에 여러 참여자]
users (1) ──── (N) challenge_checkins       [사용자 1명이 여러 인증 업로드]
challenges (1) ─── (N) challenge_checkins   [챌린지 1개에 여러 인증]
```

### 6-3. 인덱스 전략

```sql
-- 챌린지 목록 조회 최적화 (필터링)
CREATE INDEX idx_challenges_status ON challenges(status);
CREATE INDEX idx_challenges_category ON challenges(category);
CREATE INDEX idx_challenges_created_at ON challenges(created_at DESC);
CREATE INDEX idx_challenges_created_by ON challenges(created_by);

-- 참여자 조회 최적화
CREATE INDEX idx_participants_challenge_id ON challenge_participants(challenge_id);
CREATE INDEX idx_participants_user_id ON challenge_participants(user_id);
-- 현재 참여 중인 참여자 빠른 조회
CREATE INDEX idx_participants_active ON challenge_participants(challenge_id) WHERE left_at IS NULL;
-- 탈퇴 후 재참여 허용을 위한 Partial UNIQUE 인덱스 (현재 참여 중인 레코드만 유니크 보장)
CREATE UNIQUE INDEX idx_participants_unique_active ON challenge_participants(challenge_id, user_id) WHERE left_at IS NULL;

-- 인증 기록 조회 최적화
CREATE INDEX idx_checkins_challenge_id ON challenge_checkins(challenge_id);
CREATE INDEX idx_checkins_user_id ON challenge_checkins(user_id);
CREATE INDEX idx_checkins_date ON challenge_checkins(check_in_date);
-- 오늘 인증 여부 조회 최적화
CREATE INDEX idx_checkins_user_challenge_date ON challenge_checkins(user_id, challenge_id, check_in_date);
```

### 6-4. Supabase Storage 버킷 구성

| 버킷 이름              | 공개 여부 | 파일 경로 패턴                          | 허용 MIME 타입                    | 최대 크기 |
| ---------------------- | --------- | --------------------------------------- | --------------------------------- | --------- |
| `challenge-thumbnails` | 공개      | `{challenge_id}/thumbnail.{ext}`        | image/jpeg, image/png, image/webp | 5MB       |
| `checkin-images`       | 공개      | `{challenge_id}/{user_id}/{date}.{ext}` | image/jpeg, image/png, image/webp | 10MB      |

### 6-5. Row Level Security (RLS) 정책

| 테이블                 | 작업          | 정책                                              |
| ---------------------- | ------------- | ------------------------------------------------- |
| users                  | SELECT        | 모든 인증 사용자                                  |
| users                  | UPDATE        | 본인만 (auth.uid() = id)                          |
| challenges             | SELECT        | 모든 사용자 (status != 'hidden' 조건은 앱 레이어) |
| challenges             | INSERT        | 인증 사용자                                       |
| challenges             | UPDATE/DELETE | created_by = auth.uid() 또는 role = 'admin'       |
| challenge_participants | SELECT        | 모든 인증 사용자                                  |
| challenge_participants | INSERT        | 인증 사용자 (본인 user_id만)                      |
| challenge_participants | UPDATE        | 본인만 (left_at 업데이트)                         |
| challenge_checkins     | SELECT        | 모든 인증 사용자                                  |
| challenge_checkins     | INSERT        | 참여 중인 사용자만 (challenge_participants 참조)  |

### 6-6. 데이터베이스 트리거

| 트리거                        | 테이블                 | 이벤트                                     | 동작                                      |
| ----------------------------- | ---------------------- | ------------------------------------------ | ----------------------------------------- |
| `update_current_participants` | challenge_participants | INSERT / UPDATE(left_at)                   | challenges.current_participants 자동 증감 |
| `set_end_date`                | challenges             | INSERT / UPDATE(start_date, duration_days) | end_date 자동 계산                        |
| `update_updated_at`           | challenges, users      | UPDATE                                     | updated_at 자동 갱신                      |

| **챌린지 참여율**          | 생성된 챌린지당 평균 참여자 수                  | 평균 5명 이상       |
| **평균 인증률**            | (전체 인증 수 / 기대 인증 수) × 100             | 참여자당 60% 이상   |
| **재방문율**               | 가입 후 7일 내 재방문 사용자 비율               | 40% 이상            |
| **인증 지속률**            | 챌린지 시작 후 마지막 날까지 인증한 참여자 비율 | 30% 이상            |

**지표 측정 방법**

- MAU: `auth.users.last_sign_in_at` (Supabase Auth 내장 컬럼) 기준으로 당월 1회 이상 로그인한 사용자 수 집계. `auth.users` 테이블은 Supabase가 자동 관리하며 로그인 시마다 `last_sign_in_at`이 자동 갱신됨
- 인증률: challenge_checkins 레코드 수 / (challenge_participants 수 × 챌린지 경과 일수)
- 재방문율: `auth.users.last_sign_in_at`과 `auth.users.created_at`을 비교하여 가입 후 7일 내 재로그인한 사용자 비율 산출. 두 값의 차이가 1일 이상이면 재방문으로 간주
